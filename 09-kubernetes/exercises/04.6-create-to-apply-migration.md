## Overview

Lab 4.6 is about what happens when an object was created imperatively (with `kubectl create`) and you then want to manage it declaratively (with `kubectl apply`). The two commands are not equivalent. They track state differently, and the transition requires one explicit step.

The lab is a direct continuation of 4.5 (which covered the happy path of `kubectl apply` from the start). Here you intentionally start from `create` to see the gap.

## Core Concept: last-applied-configuration

`kubectl apply` manages objects by writing the full manifest into an annotation on the object itself:

```
kubectl.kubernetes.io/last-applied-configuration
```

This annotation is what enables the 3-way merge. When you run `apply` again, Kubernetes compares:
1. What was in the annotation (what you last applied)
2. What is in the cluster right now (live state)
3. What is in your file (what you want)

And it computes a diff to figure out what to add, change, or remove.

`kubectl create` does **not** write this annotation. It just creates the object. There is no recorded "previous intent" - Kubernetes has no baseline to diff against.

---

## What the Warning Looks Like

When you run `kubectl apply` on an object that was first created with `kubectl create`, you get this:

```
Warning: resource pods/nginx-pod is missing the kubectl.kubernetes.io/last-applied-configuration annotation which is required by kubectl apply. kubectl apply should only be used on resources created by either kubectl create --save-config or kubectl apply. The missing annotation will be patched automatically.
```

Key parts:
- "missing the kubectl.kubernetes.io/last-applied-configuration annotation" - confirms the object has no tracking baseline
- "should only be used on resources created by either kubectl create --save-config or kubectl apply" - `--save-config` is the flag that forces `create` to also write the annotation
- "will be patched automatically" - Kubernetes backfills the annotation from your file. After this first `apply`, the object is fully declaratively managed.

This warning is expected. It is not an error. The object is created/updated successfully.

The Pod shows `configured` (existing object updated + annotation backfilled). The Service shows `created` (new object, created declaratively from the start).


Now it exists. The object is now under declarative management.

### Step 5: Teardown


## Key Differences: create vs apply

| Behavior | `kubectl create` | `kubectl apply` |
|----------|------------------|-----------------|
| Writes `last-applied-configuration` | No | Yes |
| Idempotent (safe to re-run) | No - fails if object exists | Yes - updates if exists |
| Tracks change history for 3-way merge | No | Yes |
| Good for one-shot imperative ops | Yes | Overkill |
| Good for declarative GitOps workflow | No | Yes |
| Migrating an existing `create`-d object | First apply triggers warning | Warning is expected, annotation backfilled automatically |

### `kubectl create --save-config`

There is a flag to make `create` write the annotation:

```bash
kubectl create -f nginx-pod.yaml --save-config
```

This creates the object AND writes the `last-applied-configuration` annotation. The object is then immediately ready for `apply` without the warning. In practice most people skip this and just always use `apply` from the start.

---

## Iron Rules

1. `kubectl create` does not write `last-applied-configuration`. If you later run `apply`, you get the warning. This is expected.
2. The warning resolves itself - Kubernetes backfills the annotation from your file on the first apply. After that the object is fully declaratively managed.
3. Use `kubectl create --save-config` if you must use `create` but want to stay apply-compatible from the start.
4. In day-to-day work: just use `apply` from the beginning. Skip `create` for anything you plan to manage long-term.
5. Old ReplicaSets at 0 replicas are not mistaks - they are rollback history. Do not delete them manually unless you know you will never need to roll back.
6. Each lab lives in its own folder. Never mix YAML files between labs. Isolation prevents `kubectl apply -f .` from touching objects it shouldn't.
