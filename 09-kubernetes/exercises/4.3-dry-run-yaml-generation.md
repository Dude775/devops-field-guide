## Overview

This lab is about using `--dry-run=client -o yaml` to generate Kubernetes manifest files from imperative commands - without actually creating anything in the cluster.

The technique bridges the gap between imperative and declarative workflows. Instead of writing YAML from scratch, you let `kubectl` generate it, then apply the file declaratively. It's one of the most practical day-to-day patterns in K8s operations.

## Core Concept: Imperative-to-Declarative Bridge

The manual way to write YAML is error-prone and slow, especially when you're new to the schema. The dry-run pattern solves this:

```
kubectl run <name> --image=<img> --dry-run=client -o yaml > pod.yaml
```

You describe *what you want* using the imperative syntax you already know, and `kubectl` gives you the exact YAML that represents that intent. You review it, adjust it, commit it, and apply it declaratively.

This is also the correct exam technique for CKA/CKAD - generating manifests this way is faster than writing from memory.

---

## Key Flags

| Flag | What it does |
|------|--------------|
| `--dry-run=client` | Simulates the command client-side only. Nothing is sent to the API server. No object is created. |
| `--dry-run=server` | Sends the request to the API server for validation but doesn't persist the object. More thorough than client-side. |
| `-o yaml` | Outputs the generated manifest in YAML format to stdout. |
| `> file.yaml` | Shell redirection - captures stdout into a file. |

> `--dry-run=client` is the common exam and day-to-day variant. Use `--dry-run=server` when you need API-level validation (e.g. checking admission webhooks, CRD schema).

## Commands Practiced

### Part 1 - Generating Pod YAML via dry-run

```bash
# Preview the YAML without saving - see what kubectl would create
kubectl run color-api-pod \
  --image=lironefitoussi/color-api:1.0.0 \
  --dry-run=client -o yaml

# Save the generated manifest to a file
kubectl run color-api-pod \
  --image=nginx:alpine \
  --dry-run=client -o yaml > color-api-pod.yaml

# Verify the file before applying (Iron Rule: read before apply)
cat color-api-pod.yaml | wc -l
cat color-api-pod.yaml

# Apply declaratively from the generated file
kubectl apply -f color-api-pod.yaml

# Verify the Pod is running
kubectl get pods color-api-pod -o wide
```

### Part 2 - Generating Service YAML via dry-run

```bash
# Generate Service manifest from existing Pod (nginx-pod must exist with label app=nginx)
kubectl expose pod nginx-pod \
  --type=NodePort \
  --port=80 \
  --dry-run=client -o yaml

# Save to file
kubectl expose pod nginx-pod \
  --type=NodePort \
  --port=80 \
  --dry-run=client -o yaml > nginx-svc-dry-run.yaml

# Verify before applying
cat nginx-svc-dry-run.yaml
```

> Note: `kubectl expose` reads the target Pod's existing labels to auto-populate the Service's `selector`. The Pod must exist in the cluster for this to work - even with `--dry-run=client`, kubectl needs the Pod's metadata locally.

## What the Generated YAML Looks Like

### Pod - generated output

Running `kubectl run color-api-pod --image=nginx:alpine --dry-run=client -o yaml` produces:

```yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    run: color-api-pod
  name: color-api-pod
spec:
  containers:
  - image: nginx:alpine
    name: color-api-pod
    resources: {}
  dnsPolicy: ClusterFirst
  restartPolicy: Always
status: {}
```

### Service - generated output

Running `kubectl expose pod nginx-pod --type=NodePort --port=80 --dry-run=client -o yaml` produces:

```yaml
apiVersion: v1
kind: Service
metadata:
  labels:
    app: nginx
  name: nginx-pod
spec:
  ports:
  - port: 80
    protocol: TCP
    targetPort: 80
  selector:
    app: nginx
  type: NodePort
status:
  loadBalancer: {}
```

### Key observations

| Observation | Details |
|-------------|---------|
| `kubectl run` auto-label | Adds `run: <pod-name>` as the label - not `app:`. If you need `app:`, add `--labels=app=<name>` or edit after generating. |
| `kubectl expose` copies selector | Reads the target Pod's labels and uses them as the Service selector. That's why the Pod must exist in the cluster. |
| Service name defaults to Pod name | `kubectl expose pod nginx-pod` produces a Service named `nginx-pod`. Override with `--name=<custom-name>`. |
| Noise fields | `resources: {}`, `status: {}`, `loadBalancer: {}` are serializer defaults. Safe to delete before committing. |
| Same structure as hand-written YAML | The generated output matches what you'd write manually in labs 4.1 and 4.2 - just with extra empty fields. |

## Step by Step

### 1. Preview the YAML

```bash
kubectl run color-api-pod \
  --image=lironefitoussi/color-api:1.0.0 \
  --dry-run=client -o yaml
```

Nothing is created. You see the manifest on stdout. This is your chance to verify the structure before saving anything.

### 2. Redirect to a file

```bash
kubectl run color-api-pod \
  --image=nginx:alpine \
  --dry-run=client -o yaml > color-api-pod.yaml
```

The `>` redirects stdout into the file. The terminal shows nothing - the output went to the file.

### 3. Verify the file contents

```bash
cat color-api-pod.yaml | wc -l   # sanity check: file isn't empty
cat color-api-pod.yaml            # read the actual content
```

This is the Iron Rule: **always read the file before applying it**. Redirect bugs (e.g. wrong flags) can produce garbage that still has a `.yaml` extension.

### 4. Apply declaratively

```bash
kubectl apply -f color-api-pod.yaml
# pod/color-api-pod created
```

Using `kubectl apply` instead of `kubectl create` because `apply` is idempotent - running it again updates, not errors. Pairs naturally with the generate-then-apply workflow.

### 5. Verify

```bash
kubectl get pods color-api-pod -o wide
# NAME             READY   STATUS    RESTARTS   AGE   IP            NODE
# color-api-pod    1/1     Running   0          10s   10.244.0.21   minikube
```

## The Full Workflow Pattern

```
1. kubectl run <name> --image=<img> [flags] --dry-run=client -o yaml > manifest.yaml
2. cat manifest.yaml           # read it
3. edit manifest.yaml          # adjust fields (resources, env, labels...)
4. kubectl apply -f manifest.yaml
5. kubectl get pods <name> -o wide
6. kubectl describe pod <name> # confirm actual == desired
```

This pattern works beyond Pods. The same `--dry-run=client -o yaml` works for Deployments, Services, ConfigMaps, Jobs, and more:

```bash
# Generate a Deployment manifest
kubectl create deployment my-app --image=nginx:alpine --replicas=3 \
  --dry-run=client -o yaml > my-app-deployment.yaml

# Generate a Service manifest
kubectl expose pod color-api-pod --port=80 --target-port=8080 \
  --dry-run=client -o yaml > color-api-service.yaml
```

## Gotchas

### Image not found on Docker Hub

The instructor's image (`lironefitoussi/color-api:1.0.0`) and the backup (`lm-academy/color-api:1.0.0`) both failed with `ErrImagePull` - neither existed for `linux/amd64`.

**Fix:** don't waste time debugging an image you don't control. Swap to a known-good image (`nginx:alpine`) and focus on the concept being taught. The dry-run pattern works the same regardless of image.

### CrashLoopBackOff with hashicorp/http-echo

`hashicorp/http-echo:0.2.3` pulled fine but crashed immediately. The image requires CLI arguments to start (`-text="hello world"`). Running it bare gives a non-zero exit.

**Fix:** if a container needs args, add them to the manifest under `spec.containers[].args`. Or use a simpler image that runs without configuration.

### kubectl expose requires the Pod to exist

`--dry-run=client` still needs to read the Pod's labels from the cluster. If the Pod doesn't exist, `kubectl expose` fails even with dry-run:

```
Error from server (NotFound): pods "nginx-pod" not found
```

**Fix:** create the Pod first (`kubectl apply -f pod.yaml`), then run the expose dry-run.

### Service name defaults to Pod name

`kubectl expose pod nginx-pod ...` creates a Service named `nginx-pod` - same as the Pod. This is almost never what you want in production. Use `--name` to set a real name:

```bash
kubectl expose pod nginx-pod --name=nginx-svc --type=NodePort --port=80 --dry-run=client -o yaml
```

## Part 3 - Why This Pattern Matters

| Use case | Why dry-run helps |
|----------|------------------|
| Don't remember the YAML schema | Let `kubectl` generate it - the output is always correct |
| Starting point for a new manifest | Generate the base, then add env vars, volumes, probes, limits |
| CKA/CKAD exam | Faster than writing from scratch under time pressure |
| Onboarding to a new resource type | See the full field structure before editing |
| Reviewing what a command would produce | Preview without side effects |

The pattern is: **generate, review, adjust, apply**. The generated YAML is a first draft, not a final answer. You still need to understand what each field does to know what to keep, remove, or add.

## Iron Rules

1. `--dry-run=client` sends nothing to the API server. Nothing is created. It's safe to run anywhere, any number of times.
2. Always `cat` the file before `kubectl apply`. Redirecting to YAML doesn't mean the YAML is valid or what you expected.
3. `-o yaml` goes to **stdout**. If you forget `>`, nothing is saved and nothing breaks - you just see it on screen.
4. The noise fields (`creationTimestamp: null`, `resources: {}`, `status: {}`) are harmless - clean them out if you want a readable manifest for version control.
5. Use `kubectl apply`, not `kubectl create`, after dry-run generation. `apply` is idempotent and tracks last-applied config. `create` fails if the object already exists.
6. `kubectl run` generates `run: <name>` labels, not `app: <name>`. If a Service needs to select this Pod, either add `--labels=app=<name>` at generation time or edit the manifest before applying.
7. If an image fails with `ErrImagePull` and you don't control that image, swap to `nginx:alpine` and move on - don't debug someone else's broken registry.
