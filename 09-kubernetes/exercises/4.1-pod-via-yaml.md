## Overview

This lab is about writing your first Kubernetes manifest and using it to create a Pod with `kubectl create -f`. Not `kubectl run`, not `kubectl apply` - specifically `kubectl create -f`, which is imperative management *with* a config file.

The distinction matters because a lot of people hear "config file" and assume declarative. It's not. `kubectl create -f` is still imperative - it just reads the desired state from a file instead of command-line flags.

## Imperative Management: Two Flavors

Before writing any YAML, it's worth clarifying that "imperative" covers two different workflows:

| Flavor | Example | Notes |
|--------|---------|-------|
| Direct commands | `kubectl run nginx-pod --image=nginx:1.27.0` | No file, no YAML |
| Imperative with file | `kubectl create -f nginx-pod.yaml` | File, but still imperative |

Both tell Kubernetes *what to do right now*. Neither tracks state. The difference is just whether you type the spec inline or in a file.

Declarative (`kubectl apply -f`) is the third approach - it tracks last-applied config, enables 3-way merge, and is idempotent. That's a separate topic.

## The Manifest

`nginx-pod.yaml`:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
# annotations:
#   description: "This is a simple nginx pod for demonstration purposes."
spec:
  containers:
    - name: nginx-container
      image: nginx:1.27.0
      ports:
        - containerPort: 80
```

### Field breakdown

| Field | Value | Why it matters |
|-------|-------|----------------|
| `apiVersion` | `v1` | Pod is a core API object - lives in `v1`, not `apps/v1` |
| `kind` | `Pod` | Object type |
| `metadata.name` | `nginx-pod` | What shows up in `kubectl get pods` |
| `metadata.labels` | `app: nginx` | How Services find this Pod via selector |
| `spec.containers` | list | Required field - Pod with no containers is invalid |
| `containers[].name` | `nginx-container` | Name within the Pod spec |
| `containers[].image` | `nginx:1.27.0` | Pinned version, not `:latest` |
| `containers[].ports[].containerPort` | `80` | Informational - doesn't actually open the port, just documents it |

> Note: annotations were commented out and left out of the final manifest. Labels are for selectors and filtering. Annotations are for metadata that tools read - not for selection. In this lab there was nothing tool-specific to add, so annotations stayed empty.

## Commands

| Command | What it does |
|---------|-------------|
| `minikube start` | Start the local cluster (needs Docker Desktop running) |
| `kubectl config current-context` | Confirm you're on minikube, not a prod cluster |
| `kubectl config get-contexts` | List all available contexts |
| `kubectl cluster-info` | Check if API server is reachable |
| `kubectl get nodes` | Confirm the node is Ready |
| `kubectl run nginx-pod --image=nginx:1.27.0` | Imperative direct - create a Pod without YAML |
| `kubectl delete pod nginx-pod` | Delete a Pod (imperative) |
| `kubectl expose pod nginx-pod --port=80 --target-port=80 --name=nginx-service` | Expose Pod as a Service (imperative) |
| `kubectl create -f nginx-pod.yaml` | Imperative-with-file - create from manifest |
| `kubectl get pods` | List Pods |
| `kubectl get pods -o wide` | List with IP + Node columns |
| `kubectl describe pod nginx-pod` | Full state - verify actual vs desired |
| `docker ps` | Check running containers - if empty, minikube node may be down |

## Step by Step

### 1. Pre-flight

```bash
# confirm active context
kubectl config current-context
# should return: minikube

# confirm cluster is up
kubectl cluster-info
# if connection refused - see troubleshooting below

# check nodes
kubectl get nodes
# NAME       STATUS   ROLES           AGE
# minikube   Ready    control-plane   3d
```

### 2. Write the manifest

Create `nginx-pod.yaml` with the content above. VS Code with the Kubernetes extension gives you autocomplete, schema validation, and inline docs as you type. Worth installing.

### 3. Create the Pod

```bash
kubectl create -f nginx-pod.yaml
# pod/nginx-pod created
```

### 4. Verify it came up

```bash
kubectl get pods
# NAME        READY   STATUS    RESTARTS   AGE
# nginx-pod   1/1     Running   0          15s

kubectl get pods -o wide
# NAME        READY   STATUS    RESTARTS   AGE   IP             NODE
# nginx-pod   1/1     Running   0          30s   10.244.0.20    minikube
```

### 5. Confirm actual state matches desired state

```bash
kubectl describe pod nginx-pod
```

Key fields to verify:

| Field | Expected |
|-------|----------|
| Name | nginx-pod |
| Labels | app=nginx |
| Annotations | none |
| Status | Running |
| IP | 10.244.0.20 (will vary) |
| Container name | nginx-container |
| Image | nginx:1.27.0 |
| Port | 80/TCP |
| Node | minikube |

If these match your manifest, the lab is done.

## Troubleshooting: connection refused on API server

```
The connection to the server 127.0.0.1:PORT was refused - did you specify the right host or port?
```

This is not a kubectl config bug. It means the API server is down. Most likely: minikube stopped.

**Diagnosis:**

```bash
docker ps
# If empty or no minikube container → cluster is down
```

**Fix:**

```bash
# 1. Make sure Docker Desktop is running
# 2. Then:
minikube start
```

minikube needs Docker running first because it uses Docker as the VM driver. The sequence matters.

## Iron Rules

1. `kubectl create -f` = imperative management WITH a config file - not declarative. Declarative is `kubectl apply`.
2. `spec.containers` is required for any Pod. Missing it = validation error. It's a list, not a single object.
3. `kubectl describe` is your proof that actual state == desired state. Don't assume the Pod is correct just because it's Running.
4. `connection refused` on the API server usually means the cluster isn't running - not a config bug. Check `docker ps` first.

## Anchors to Other Modules

- **07-docker**: `nginx:1.27.0` is the same Docker image artifact. A Pod wraps one or more containers - same image, same concept, just managed by K8s instead of `docker run`.
- **07-docker**: `containerPort: 80` maps directly to `EXPOSE 80` in a Dockerfile. Same idea, different layer.
- **YAML**: the manifest structure echoes docker-compose service definitions - `name`, `image`, `ports` are familiar fields.
