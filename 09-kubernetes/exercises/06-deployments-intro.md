## Why Deployments Exist

ReplicaSets have one job and do it well - but that job is too narrow for production:

| Gap | ReplicaSet | Deployment |
|-----|-----------|------------|
| Image update | Manual Pod delete required | Automatic rolling update |
| Rollback | No history | `kubectl rollout undo` |
| Update history | None | Up to `revisionHistoryLimit` (default 10) |
| Zero-downtime | DIY | Built-in with RollingUpdate strategy |
| Pause/resume rollout | Not possible | `kubectl rollout pause/resume` |

A Deployment manages ReplicaSets. You write one Deployment. Kubernetes creates and manages the ReplicaSets for you.

---

## The Three-Layer Hierarchy

```
Deployment   <- you write and manage this
    |
    v creates automatically
ReplicaSet   <- never touch this manually
    |
    v creates automatically
Pods         <- actual running containers
```

When you change the Deployment template (image, env vars, etc.), Kubernetes:
1. Creates a NEW ReplicaSet with the updated template
2. Scales it up gradually
3. Scales down the OLD ReplicaSet gradually
4. Keeps the old RS at 0 replicas (for instant rollback)

---

## pod-template-hash - The Glue

Every RS created by a Deployment gets a label: `pod-template-hash: <hash>`

The hash is computed from the Pod template content. If the template changes, the hash changes, so a new RS is created.

Pods inherit this label:
```
nginx-deployment-6d8c4f7b5c-xkj9m
               ^^^^^^^^^^           <- RS hash
                            ^^^^^   <- random pod suffix
```

Pod name = `<deployment-name>-<rs-hash>-<random>`. The whole lineage is visible in the name.

**Never set `pod-template-hash` manually** - it's auto-generated by the Deployment controller.

## Rolling Update Strategy

Default strategy. Controlled by two fields:

```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 25%        # max Pods above desired count during update
    maxUnavailable: 25%  # max Pods below desired count during update
```

For 4 replicas with 25%/25%:
- Can have up to 5 Pods running at once (4 + 1)
- Can have as few as 3 Pods running at once (4 - 1)

**During a rolling update, TWO ReplicaSets are alive simultaneously:**

```
Before:  RS-v1: 4 pods, RS-v2: 0 pods
Step 1:  RS-v1: 3 pods, RS-v2: 1 pod   <- +1 new, -1 old
Step 2:  RS-v1: 2 pods, RS-v2: 2 pods
Step 3:  RS-v1: 1 pod,  RS-v2: 3 pods
After:   RS-v1: 0 pods, RS-v2: 4 pods  <- old RS preserved at 0
```

The Service routes to ALL matching Pods during the rollout. Users may hit v1 or v2 during the transition.

**The `strategy` field lives in the Deployment spec, NOT in the ReplicaSet.** When RS objects are created, they don't carry the strategy - the Deployment controller orchestrates it.

## Deployment YAML

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 5
  selector:
    matchLabels:
      app: nginx
  strategy:
    type: RollingUpdate          # spec level, NOT inside template.spec
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
        - name: nginx
          image: nginx:1.27.0
          ports:
            - containerPort: 80
```

**Common indentation mistake**: putting `strategy` inside `template.spec`. It belongs at `spec` level (sibling of `replicas`, `selector`, `template`).

## Rollout Commands

```bash
# watch rollout progress live
kubectl rollout status deployment/nginx-deployment

# list revision history
kubectl rollout history deployment/nginx-deployment

# rollback to previous version
kubectl rollout undo deployment/nginx-deployment

# rollback to specific revision
kubectl rollout undo deployment/nginx-deployment --to-revision=2

# pause a rollout mid-way
kubectl rollout pause deployment/nginx-deployment

# resume a paused rollout
kubectl rollout resume deployment/nginx-deployment

# tag a revision with a message (for history readability)
kubectl annotate deployment/nginx-deployment kubernetes.io/change-cause="updated to v2"
```

## Iron Rules

- Always use `kind: Deployment`. Never raw ReplicaSet in production.
- `strategy` field is at `spec` level, not inside `template.spec`.
- Default strategy is RollingUpdate - it's active even without an explicit `strategy:` block.
- Old RS is preserved at 0 replicas after update - that's your rollback lever.
- `pod-template-hash` is auto-added by Kubernetes. Never set it manually.
- Pod name format `<deployment>-<rs-hash>-<random>` reveals the full lineage.

> "If Kubernetes manages versions, why do we need manual rollback? Can't it detect broken code automatically?"

Short answer: it can, but it needs help. This involves:
- **Liveness Probe** - is the container still alive? (kills and restarts if not)
- **Readiness Probe** - is the container ready to receive traffic? (removes from Service if not)
- `progressDeadlineSeconds` - fail the rollout if it doesn't finish in time
- **Argo Rollouts** - canary/blue-green with automated analysis
- Observability stack (metrics, error rates) as rollout signals

## Interview Questions

- "How does a Deployment achieve zero-downtime updates?"  
  Answer: RollingUpdate creates a new RS and gradually shifts Pods from old RS to new. maxSurge/maxUnavailable control the pace.

- "What happens if you change both `replicas` AND `image` in the same apply?"  
  Answer: Kubernetes creates a new RS for the new image, scales it to the new replica count. One operation, two changes reconciled together.

- "Why does Kubernetes need ReplicaSet if Deployment exists?"  
  Answer: Deployment is an orchestrator of RS versions. The RS does the actual Pod lifecycle management. Separation of concerns - RS is a stable primitive, Deployment adds versioning on top.
